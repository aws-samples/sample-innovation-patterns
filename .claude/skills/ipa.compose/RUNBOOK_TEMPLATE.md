# Runbook Generation Template

> Reference file for `/ipa.compose` SKILL.md. Loaded on demand during runbook generation.
> Contains section structure and content patterns for `docs/infra/runbook.md`.

**Critical rule**: The runbook must be self-contained. No references to IPA skills, Claude Code, or the AI agent. A customer engineer reads this document to deploy, verify, and tear down infrastructure independently.

---

## Section 1: Title and Overview

```markdown
# Infrastructure Runbook: {Pattern Name}

> Generated by /ipa.compose on {YYYY-MM-DD}. This document is the deployment guide
> for {APP_NAMESPACE}-{APP_ENV} infrastructure. Follow these instructions to
> deploy, verify, and tear down the infrastructure without IPA tooling.

## Architecture

{Copy the full content of the pattern's ARCHITECTURE.md here — diagram and stack-by-stack summary}

### Stack Summary

| Stack | Service | Stack Name | Template |
|-------|---------|------------|----------|
```

Populate the Stack Summary table from the stack inventory (one row per stack in deployment order).

Use the `.env` variable convention for stack names: `$(APP_NAMESPACE)-$(APP_ENV)-{suffix}`.

## Section 1c: Added Pattern (Pattern-Level Composition)

> **When to use**: Include this subsection when a builder adds a second (or subsequent) pattern on top of an existing composition (add-pattern mode). This is distinct from add-stack mode — a pattern brings its own architecture narrative, not just individual stacks. This section is appended after the base pattern's architecture content, inside the Architecture section.
>
> - The content comes from the added pattern's `patterns/{name}/ARCHITECTURE.md` and is inserted verbatim.
> - If multiple patterns are added over time, each gets its own "Added Pattern" section, in the order they were added.
> - This section captures the architectural intent of the added pattern as a whole; individual stack details are already covered by the Stack Summary table and any Added Stacks subsections.

```markdown
### Added Pattern: {pattern-name}

*Added to {base-pattern} composition on {YYYY-MM-DD}.*

{Full content from the added pattern's ARCHITECTURE.md, inserted verbatim.}
```

Repeat the `### Added Pattern: {pattern-name}` block for each pattern added to the composition. Order matches the chronological order of addition.

## Section 5: Security Setup

```markdown
## Security Setup

Before deploying application stacks, ensure the security stack is deployed:
```bash
# Verify security stack exists
uv run --project utils deploy cfn-status --stack-name $(APP_NAMESPACE)-$(APP_ENV)-security
```

Expected status: `CREATE_COMPLETE` or `UPDATE_COMPLETE`.

If not deployed, deploy the security CloudFormation template manually:
```bash
aws cloudformation deploy \
  --template-file infra/cfn/generated/security.yml \
  --stack-name $(APP_NAMESPACE)-$(APP_ENV)-security \
  --capabilities CAPABILITY_NAMED_IAM \
  --region $(AWS_REGION) \
  --profile $(AWS_PROFILE)
```
```

## Section 5a: Prepare (One-Time Prerequisites)

> **Always include** this section since prepare.mk is always generated. When no prepare stacks exist, write: "No prepare prerequisites for this pattern." and omit the table, verify, and teardown subsections.

```markdown
## Prepare (One-Time)

> Run once before first deployment. Re-run when prepare targets change
> (e.g., new stack added to the pattern).

```bash
make -f scripts/prepare.mk prepare
```

### What This Deploys

| Stack | Service | Stack Name |
|-------|---------|------------|
```

Populate the table from prepare stacks (one row per prepare stack).

```markdown

### Verify Prepare Stacks

```bash
uv run --project utils deploy cfn-status --stack-name $(APP_NAMESPACE)-$(APP_ENV)-{suffix}
```

Expected status: `CREATE_COMPLETE` or `UPDATE_COMPLETE`.

### When to Re-Run

- After adding new stacks that require prepare infrastructure
- After changing prepare stack parameters
- NOT needed on every deployment — prepare stacks persist across deploy/destroy cycles

### Teardown (Manual Only)

Prepare stacks are NOT deleted by the standard teardown process. To manually remove:

```bash
make -f scripts/prepare.mk teardown-prepare
```

> **Warning**: Tearing down prepare stacks (e.g., ECR) may require emptying
> the resource first (e.g., deleting all container images from ECR).
```

## Section 9: Teardown

```markdown
## Teardown

> **IMPORTANT**: Delete stacks in REVERSE deployment order to avoid
> CloudFormation cross-stack reference errors.

```bash
make -f scripts/deploy.mk teardown
```

Or delete individually in reverse order:

1. Delete {StackN}: `make -f scripts/deploy.mk teardown-{suffixN}`
2. Verify: `uv run --project utils deploy cfn-status --stack-name $(APP_NAMESPACE)-$(APP_ENV)-{suffixN}`
   (expected: stack not found)
3. Delete {StackN-1}: `make -f scripts/deploy.mk teardown-{suffixN-1}`
{... continue in reverse order}
```

## Section 10: Troubleshooting

Always include these three failure scenarios:

```markdown
## Troubleshooting

### ROLLBACK_COMPLETE

**Symptom**: Stack status is ROLLBACK_COMPLETE.
**Cause**: Initial creation failed; CloudFormation rolled back.
**Fix**: Delete the failed stack and retry.
```bash
uv run --project utils deploy cfn-delete --stack-name {stack-name}
# Wait for deletion, then re-run the deploy target
```

### Permission Denied

**Symptom**: CREATE_FAILED with "not authorized to perform" error.
**Cause**: Missing IAM permission on the execution role.
**Fix**: Check the security stack's execution role has the required permissions. Update the security configuration and re-deploy.

### Resource Already Exists

**Symptom**: CREATE_FAILED with "already exists" error.
**Cause**: A resource with the same name exists outside this stack.
**Fix**: Change APP_NAMESPACE in .env to generate different resource names, then re-deploy.
```
